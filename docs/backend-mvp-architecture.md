# Backend MVP Architecture and Scope

Issue: #1

## 1. Product Goal

Build a Facebook-like demo backend that supports a small social network experience: user accounts, profiles, posts, comments, reactions, friendships or following, notifications, and a simple personalized feed. The MVP should favor clear backend boundaries, predictable APIs, and straightforward data modeling over production-scale social graph complexity.

## 2. MVP Scope

### In Scope

- User registration, login, logout, and authenticated API access.
- Profile creation and profile updates for display name, bio, avatar URL, and basic public metadata.
- Post creation, editing, deletion, listing, and visibility rules for public and friends-only posts.
- Comment creation, editing, deletion, and listing per post.
- Reactions on posts and comments, initially limited to a small fixed set such as `like`, `love`, and `laugh`.
- Friend request lifecycle: send, accept, reject, cancel, and unfriend.
- Feed generation for the authenticated user from their own posts and accepted friends' posts.
- Notification records for friend requests, accepted requests, comments, and reactions.
- Basic moderation controls through soft deletes and ownership checks.
- API documentation and automated tests for core flows.

### Out of Scope for MVP

- Real-time chat and typing indicators.
- Marketplace, groups, pages, events, stories, reels, livestreaming, and advertising.
- Complex ranking, machine-learning recommendations, and large-scale graph processing.
- Full-text search beyond simple user or post lookup.
- Media upload processing; the MVP can store externally hosted media URLs.
- Multi-tenant administration and advanced role management.

## 3. Technical Decisions

- Backend style: RESTful API with versioned routes under `/api/v1`.
- Runtime: Node.js with a modular service architecture, or the existing repository default if a framework is later selected.
- Database: PostgreSQL for relational integrity across users, posts, comments, reactions, friendships, and notifications.
- ORM or query layer: use the repository's chosen standard once implementation starts; prefer migrations checked into source control.
- Authentication: short-lived access tokens plus refresh tokens stored and rotated server-side, with password hashing through a proven adaptive hash such as bcrypt or Argon2.
- Authorization: resource ownership and friendship visibility checks enforced in service-level policies.
- Validation: request DTO/schema validation at the API boundary.
- Error format: consistent JSON error responses with stable machine-readable codes.
- Observability: structured logs, request IDs, health checks, and basic metrics hooks.
- Testing: unit tests for services and policies, integration tests for API and persistence flows.

## 4. Module Responsibilities

### Auth Module

- Register users and normalize credentials.
- Authenticate users and issue token pairs.
- Refresh and revoke sessions.
- Hash and verify passwords.
- Provide middleware or guards that attach the authenticated user context.

### Users and Profiles Module

- Store account identity and profile metadata.
- Expose authenticated self-profile operations.
- Expose public profile lookup with privacy-safe fields.
- Enforce username/email uniqueness and profile update validation.

### Social Graph Module

- Manage friend requests and accepted friendships.
- Prevent duplicate, circular, or invalid requests.
- Provide friendship status checks for other modules.
- Support friend list retrieval.

### Posts Module

- Create, update, soft-delete, and retrieve posts.
- Enforce visibility rules and author ownership.
- Store optional media URLs and post metadata.
- Provide post listings by author and by feed candidate query.

### Comments Module

- Create, update, soft-delete, and list comments for posts.
- Enforce post visibility before comment access.
- Enforce comment ownership for edits and deletes.
- Emit notification events when a user comments on another user's post.

### Reactions Module

- Add, change, remove, and count reactions.
- Ensure each user can have only one reaction per target.
- Support reactions on posts and comments.
- Emit notification events for reactions on another user's content.

### Feed Module

- Build the authenticated user's home feed from self and accepted friends' posts.
- Apply visibility filters before returning content.
- Sort by creation time for MVP; leave ranking pluggable for later.
- Paginate with cursor-based pagination to avoid offset drift.

### Notifications Module

- Store notification records generated by domain events.
- List unread and recent notifications for the authenticated user.
- Mark notifications as read.
- Avoid creating notifications for self-actions.

### Shared Infrastructure Module

- Configuration loading and environment validation.
- Database connection and transaction helpers.
- Request logging, request IDs, and error handling.
- Common pagination, validation, and response helpers.

## 5. API Strategy

All endpoints should be versioned under `/api/v1`. Responses should use JSON and include stable IDs, ISO-8601 timestamps, and pagination metadata where applicable.

### Auth

- `POST /api/v1/auth/register`
- `POST /api/v1/auth/login`
- `POST /api/v1/auth/refresh`
- `POST /api/v1/auth/logout`

### Users and Profiles

- `GET /api/v1/users/me`
- `PATCH /api/v1/users/me`
- `GET /api/v1/users/{userId}`
- `GET /api/v1/users/{userId}/friends`

### Friendships

- `POST /api/v1/friend-requests`
- `GET /api/v1/friend-requests/incoming`
- `GET /api/v1/friend-requests/outgoing`
- `POST /api/v1/friend-requests/{requestId}/accept`
- `POST /api/v1/friend-requests/{requestId}/reject`
- `DELETE /api/v1/friend-requests/{requestId}`
- `DELETE /api/v1/friends/{friendUserId}`

### Posts, Comments, and Reactions

- `POST /api/v1/posts`
- `GET /api/v1/posts/{postId}`
- `PATCH /api/v1/posts/{postId}`
- `DELETE /api/v1/posts/{postId}`
- `GET /api/v1/users/{userId}/posts`
- `POST /api/v1/posts/{postId}/comments`
- `GET /api/v1/posts/{postId}/comments`
- `PATCH /api/v1/comments/{commentId}`
- `DELETE /api/v1/comments/{commentId}`
- `PUT /api/v1/posts/{postId}/reaction`
- `DELETE /api/v1/posts/{postId}/reaction`
- `PUT /api/v1/comments/{commentId}/reaction`
- `DELETE /api/v1/comments/{commentId}/reaction`

### Feed and Notifications

- `GET /api/v1/feed`
- `GET /api/v1/notifications`
- `POST /api/v1/notifications/{notificationId}/read`
- `POST /api/v1/notifications/read-all`

## 6. Data Model Overview

### users

- `id` primary key
- `email` unique, required
- `username` unique, required
- `password_hash` required
- `status` active, disabled, deleted
- `created_at`, `updated_at`

### profiles

- `user_id` primary key and foreign key to users
- `display_name`
- `bio`
- `avatar_url`
- `cover_url`
- `created_at`, `updated_at`

### friendships

- `id` primary key
- `requester_id` foreign key to users
- `addressee_id` foreign key to users
- `status` pending, accepted, rejected, cancelled
- `requested_at`, `responded_at`
- Unique constraint on the normalized user pair to prevent duplicate active relationships.

### posts

- `id` primary key
- `author_id` foreign key to users
- `body`
- `visibility` public, friends
- `media_urls` JSON array or child table, depending on implementation preference
- `deleted_at`
- `created_at`, `updated_at`

### comments

- `id` primary key
- `post_id` foreign key to posts
- `author_id` foreign key to users
- `body`
- `deleted_at`
- `created_at`, `updated_at`

### reactions

- `id` primary key
- `target_type` post or comment
- `target_id`
- `user_id` foreign key to users
- `reaction_type`
- `created_at`, `updated_at`
- Unique constraint on `target_type`, `target_id`, and `user_id`.

### notifications

- `id` primary key
- `recipient_id` foreign key to users
- `actor_id` nullable foreign key to users
- `type`
- `entity_type`
- `entity_id`
- `read_at`
- `created_at`

### sessions

- `id` primary key
- `user_id` foreign key to users
- `refresh_token_hash`
- `expires_at`
- `revoked_at`
- `created_at`

## 7. Non-Functional Requirements

- Security: password hashing, token rotation, rate limits for auth endpoints, ownership checks, and no sensitive fields in API responses.
- Privacy: friendship-aware visibility enforcement for posts, comments, and profile surfaces.
- Reliability: transactional writes for multi-step operations such as accepting friend requests and generating notifications.
- Performance: indexed foreign keys, cursor pagination, and bounded page sizes.
- Maintainability: module-level boundaries, migration-based schema changes, typed DTOs or schemas, and consistent error handling.
- Testability: deterministic services with mocked external dependencies and integration coverage for critical API flows.
- Observability: structured request logs, health endpoint, and explicit error codes for operational diagnosis.

## 8. Implementation Roadmap

### Phase 1: Foundation

- Select and scaffold the backend framework.
- Add configuration validation, database connection, migration tooling, and health check.
- Add global request validation, error handling, and logging.

### Phase 2: Authentication and Profiles

- Implement users, profiles, registration, login, refresh, and logout.
- Add auth guards and request user context.
- Cover auth flows with integration tests.

### Phase 3: Social Graph

- Implement friend request lifecycle and accepted friendships.
- Add relationship status checks and friend list APIs.
- Add tests for duplicate requests, self-requests, and state transitions.

### Phase 4: Content

- Implement posts, comments, reactions, ownership checks, and soft deletes.
- Enforce visibility rules across read and write paths.
- Add notification event creation for comments and reactions.

### Phase 5: Feed and Notifications

- Implement cursor-paginated home feed.
- Implement notification listing and read state.
- Add integration tests for feed visibility and notification generation.

### Phase 6: Hardening

- Add rate limiting for authentication and write-heavy endpoints.
- Add API documentation and example requests.
- Review indexes, pagination behavior, error coverage, and deployment configuration.

## 10. Open Product Decisions

- Whether relationships should be symmetric friendships only, one-way following only, or both.
- Whether comments should support replies in the MVP.
- Whether post visibility should include private posts in addition to public and friends-only.
- Whether media upload should be added to the backend or delegated to an external storage flow after MVP.
